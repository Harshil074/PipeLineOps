Keeping the pipeline cheap

Jenkins runs on my own machine, so there's no metered CI-minute bill — but AWS S3 hosting is pay-as-you-go. The habits below are what actually matter once a pipeline runs hundreds of times a day, on either kind of infrastructure.

~40% faster

Cache dependencies between runs

Restoring a cached node_modules instead of reinstalling on every run cuts several minutes off each build — time that adds up fast on a self-hosted Jenkins node running many builds a day.

Fewer runs

Path filters on triggers

Only run the full pipeline when docs/**.md or config files change if they can't break the site; skip straight past stages that can't possibly fail.

~$0/month

Static hosting over a server

AWS S3 charges for storage and requests, not compute time — for a site this size that works out to fractions of a cent, versus paying for an always-on server that sits idle most of the day.

Fail fast

Cheapest checks first

Lint and JSON validation run before the heavier headless-browser test stage, so a typo fails in seconds instead of after a full test suite spins up.

No wasted runs

Concurrency limits

Jenkins queues new builds instead of running them in parallel, so pushing three fixes in a row runs them one after another rather than paying for three overlapping pipeline runs competing for the same resources.
